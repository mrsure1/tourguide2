1. 이 사이트에 맞는 언어 정책 초안

## 1. 전제와 현재 상태

### 서비스 전제

- 주요 고객은 한국 여행을 예약하려는 외국인 여행자다.
- 공급자는 주로 한국인 가이드다.
- 따라서 `고객 획득 언어`와 `운영 언어`를 분리해서 설계해야 한다.

### 현재 코드 기준 관찰 사항

- 메인 진입점은 `/`로 통합되어 있다.
  - 파일: `app/page.tsx`
  - UI 컴포넌트: `components/home/MainLandingClient.tsx`
- 예전 여행자 홈 `/traveler/home`은 현재 `/`로 흡수하는 방향으로 정리 중이다.
  - 파일: `app/traveler/home/page.tsx`
- 전체 레이아웃의 HTML 언어는 아직 `ko`다.
  - 파일: `app/layout.tsx`
- 전역 메타데이터의 Open Graph locale도 `ko_KR`로 고정되어 있다.
  - 파일: `app/layout.tsx`
- 검색/탐색 UI는 한국어 문자열이 코드에 직접 박혀 있다.
  - 파일: `components/home/MainLandingClient.tsx`
  - 파일: `components/search/SearchClient.tsx`
- 약관/고객센터/가이드 운영 페이지는 거의 전부 한국어 기준이다.
  - 파일: `app/terms/page.tsx`
  - 파일: `app/support/page.tsx`
  - 파일: `app/guide/*`
- 언어 검색 관련 유틸은 이미 다국어 표현을 일부 고려하고 있다.
  - 파일: `lib/search/language.ts`

### 결론

현재 구조는 `한국어 운영형 서비스`에 가깝다.  
하지만 실제 타깃 고객이 외국인 여행자라면, 공개 영역은 영어 기준으로 재정렬하는 것이 맞다.

## 2. 언어 정책 초안

## 핵심 원칙

- 공개 여행자 퍼널: `영어 우선`
- 가이드 운영 퍼널: `한국어 우선`
- 관리자 운영 퍼널: `한국어 우선`
- 법률/정책 문서: `한국어 원문 + 영어 번역본 병행`
- 가이드/투어 콘텐츠: `원문 + 번역본` 구조로 저장

## 권장 기본 언어

- 비로그인 공개 사용자 기본값: `en`
- 로그인 여행자 기본값: `en`
- 로그인 가이드 기본값: `ko`
- 관리자 기본값: `ko`

## 언어 우선순위 결정 규칙

언어 결정 우선순위는 아래 순서를 권장한다.

1. 사용자가 직접 선택한 언어
2. 로그인 사용자 프로필의 `preferred_locale`
3. URL locale
4. 쿠키 locale
5. 브라우저 `Accept-Language`
6. 기본 fallback

## fallback 원칙

- 여행자용 UI fallback: `en`
- 가이드/관리자 UI fallback: `ko`
- 콘텐츠 fallback: `원문 언어 그대로 노출 + 언어 배지 표시`

예:

- 영어 페이지인데 영어 번역이 없는 가이드 소개는 한국어 원문을 보여줄 수 있다.
- 단, `Korean original` 또는 `Translated soon` 같은 배지를 함께 노출해야 한다.

## 운영 원칙

- 가이드는 한국어로 입력해도 된다.
- 여행자에게 노출되는 핵심 콘텐츠는 영어 버전을 우선 확보한다.
- 자동 번역은 보조 수단으로만 쓰고, 최종 노출용 핵심 문구는 검수된 텍스트를 권장한다.

## IA 운영 원칙 요약

- 여행자가 `탐색`, `비교`, `예약`하는 모든 화면은 영어 우선
- 가이드가 `등록`, `운영`, `정산`하는 모든 화면은 한국어 우선
- 혼합 사용자 영역은 `역할 기반 기본 언어`를 적용

## 4. 실제 코드 기준 i18n 적용 구조 제안

## 권장 방식

현재 프로젝트는 Next.js App Router 구조다.  
이 경우 `UI 번역`, `라우팅`, `메타데이터`, `서버 컴포넌트`, `클라이언트 컴포넌트`를 함께 다뤄야 한다.

권장 구조는 아래 두 단계다.

1. 1차: `dictionary + locale provider` 기반 최소 침습 도입
2. 2차: 공개 여행자 영역을 `locale prefix` 기반으로 확장

## 4-1. 1차 도입안: 현재 코드에 가장 현실적인 방식

### 목표

- 지금 있는 라우트를 최대한 유지한다.
- `/`는 영어 기본으로 바꾼다.
- `guide/admin` 영역은 한국어 기본으로 유지한다.
- 공통 UI 문자열을 코드에서 분리한다.

### 권장 파일 구조

```text
lib/
  i18n/
    config.ts
    get-request-locale.ts
    get-default-locale.ts
    dictionary.ts
    format.ts
messages/
  en/
    common.json
    landing.json
    auth.json
    booking.json
    support.json
  ko/
    common.json
    landing.json
    auth.json
    booking.json
    support.json
components/
  providers/
    LocaleProvider.tsx
  i18n/
    LanguageSwitcher.tsx
```

### 권장 주요 파일 역할

- `lib/i18n/config.ts`
  - 지원 언어 목록
  - 기본 언어
  - 역할별 기본 언어 맵

- `lib/i18n/get-request-locale.ts`
  - 쿠키
  - 사용자 프로필
  - 브라우저 언어
  - 역할
  - URL
  를 바탕으로 현재 locale 결정

- `lib/i18n/dictionary.ts`
  - 서버에서 번역 파일 로드
  - `getDictionary(locale, namespace)` 제공

- `components/providers/LocaleProvider.tsx`
  - 클라이언트 컴포넌트에서 locale/dictionary 접근

- `components/i18n/LanguageSwitcher.tsx`
  - 헤더/푸터에 공통 삽입

## 4-3. 라우팅 전략 제안

### 단기 권장

- 경로는 우선 유지
- locale은 쿠키 + 사용자 설정으로 처리

예:

- `/`는 영어 기본
- `/guide/*`는 한국어 기본
- `/admin/*`는 한국어 기본

### 중장기 권장

공개 여행자 영역은 locale prefix로 옮기는 것이 SEO에 더 유리하다.

예:

- `/en`
- `/ko`
- `/en/guides/[id]`
- `/en/tours/[id]`

하지만 현재 코드 기준으로는 전체 라우트가 이미 구성되어 있으므로, 1차는 prefix 없이 시작하고 2차에 공개 영역만 이전하는 것이 현실적이다.

## 5-3. 권장 DB 구조

현재 코드에서 자주 사용하는 컬럼은 아래다.

- `guides_detail.bio`
- `guides_detail.location`
- `tours.title`
- `tours.description`
- `tours.region`

초기 권장안은 `jsonb` 기반 다국어 필드 추가다.

예:

- `guides_detail.bio_i18n jsonb`
- `tours.title_i18n jsonb`
- `tours.description_i18n jsonb`
- `profiles.preferred_locale text`

예시:

```json
{
  "ko": "한옥과 골목 문화에 강한 서울 로컬 가이드입니다.",
  "en": "Local Seoul guide specialized in hanok neighborhoods and hidden alley culture."
}
```

### 권장 이유

- 컬럼 폭발을 막을 수 있다.
- `ja`, `zh` 추가가 쉬워진다.
- 현재 Supabase/Postgres 구조와 잘 맞는다.

## 5-4. region/location 처리 원칙

`region`과 `location`은 자유 텍스트보다 `코드 + 다국어 라벨` 구조가 더 낫다.

예:

- `region_code = "SEOUL"`
- 화면 출력은 dictionary 또는 locale label map 사용

이렇게 하면 검색/필터/SEO가 안정적이다.

```ts
dict.hero.title

```

## 7-3. 콘텐츠 조회 helper

```ts
export function pickLocalizedText(
  value: Record<string, string> | null | undefined,
  locale: string,
  fallback: string[] = ["en", "ko"],
) {
  if (!value) return "";
  if (value[locale]) return value[locale];
  for (const key of fallback) {
    if (value[key]) return value[key];
  }
  return "";
}
```

## 9. 권장 마이그레이션 순서

## Step 1

- `preferred_locale` 도입
- `messages/en`, `messages/ko` 추가
- 공통 헤더/푸터/UI 문구 분리

## Step 2

- 랜딩/검색/로그인/회원가입을 영어 우선으로 전환
- `app/layout.tsx`의 `lang`, metadata locale 동적화

## Step 3

- 가이드 상세/투어 상세의 콘텐츠 번역 필드 도입
- `bio_i18n`, `title_i18n`, `description_i18n` 적용

## Step 4

- 고객센터/FAQ/약관을 다국어 문서로 분리
- 법률 문서는 `ko 원문`, `en 번역본` 정책 고정

## Step 5

- 공개 여행자 영역 locale prefix 전환 검토
- 필요 시 `/en`, `/ko` 구조로 SEO 최적화

## 10. 최종 권장 결론

이 서비스는 아래 구조가 가장 현실적이다.

- 공개 여행자 영역: `영어 우선`
- 가이드/관리자 운영 영역: `한국어 우선`
- UI: i18n dictionary 기반
- 콘텐츠: DB 다국어 필드 기반
- 1차는 현재 라우트 유지
- 2차는 공개 페이지만 locale prefix 확장

한 문장으로 정리하면:

`고객 획득은 영어, 공급자 운영은 한국어, 구현은 단계적 i18n 구조로 분리`가 가장 적절하다.

## 11. 바로 이어서 할 수 있는 작업

문서 기준 다음 작업 우선순위는 아래다.
